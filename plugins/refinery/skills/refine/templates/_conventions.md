# Specification Conventions

This document defines the writing standards for all specifications in this project, generated by `/refine init`. Override sections as needed to match project preferences; `init` will preserve customizations on re-run.

## 1. Requirement Syntax (EARS)

Per Refinery's reference (`${CLAUDE_SKILL_DIR}/references/requirement-syntax.md`):

| Pattern | Form |
|---------|------|
| Ubiquitous | The {system} shall {behavior} |
| Event-Driven | When {trigger}, the {system} shall {behavior} |
| State-Driven | While {state}, the {system} shall {behavior} |
| Unwanted Behavior | If {unwanted condition}, then the {system} shall {behavior} |
| Optional Feature | Where {feature included}, the {system} shall {behavior} |
| Complex | Combination — use sparingly; prefer decomposition |

## 2. Acceptance Criteria (Given/When/Then)

```gherkin
Given <precondition>
  And <additional precondition>     # optional
When <action>
Then <observable outcome>
  And <additional outcome>          # optional
```

**Rules:**

- Atomic: one clear action per scenario
- Deterministic: same Given+When → same Then
- Observable: testable from outside the system (response, file, log, exception)
- Concrete: specific values, not vague qualifiers
- Independent: no inter-AC test dependencies

## 3. RFC 2119 Language (System Specs Only)

| Keyword | Meaning |
|---------|---------|
| MUST / SHALL / REQUIRED | Absolute requirement |
| MUST NOT / SHALL NOT | Absolute prohibition |
| SHOULD / RECOMMENDED | Strong recommendation; deviations require justification |
| SHOULD NOT | Strong recommendation against |
| MAY / OPTIONAL | Truly optional |

Use **uppercase** for RFC 2119 keywords. Use lowercase `shall` inside EARS patterns (per EARS standard).

## 4. Requirement IDs

| Type | Format | Scope |
|------|--------|-------|
| Functional Requirement | `FR-NNN` | Per artifact |
| Non-Functional Requirement | `NFR-<CATEGORY>-NNN` (categories: P, S, SC, R, A, U, M, C) | Per artifact |
| Acceptance Criterion | `AC-<REQ-ID>-N` | Per parent requirement |
| Invariant | `INV-NNN` | Per artifact |
| Resolved Design Decision | `RD-NNN` | Per artifact |
| Risk | `R-NNN` | Per artifact |
| Failure Mode | `FM-NNN` | Per design artifact |
| Principle | `P-NNN` | Per principles artifact |
| Open Question | `OQ-NNN` | Per artifact |
| Source Document | `SD-NNN` | Per artifact's source_documents list |
| Ticket | `T-NN` | Per tickets artifact |

**Reuse policy:** Never reuse a deleted ID. Mark deletions as `[DELETED — <reason>]` rather than removing.

## 5. Priorities (MoSCoW)

| Priority | Meaning |
|----------|---------|
| Must | Required for ship |
| Should | Expected, deferable |
| Could | Nice to have |
| Won't | Explicitly excluded |

## 6. File Naming

- Lowercase, hyphenated: `user-authentication-spec.md`
- Complex specs may use directories: `user-authentication/`
- Per-project override allowed; document choice below

## 7. Document Structure

Per Refinery's universal format (see `${CLAUDE_SKILL_DIR}/references/document-format.md`):

1. YAML frontmatter (universal fields per §1)
2. Body (artifact-type-specific sections per §3)
3. Open Questions (always present, may be empty)
4. Iteration Log (always present, append-only)
5. Changelog (always present, append-only)

## 8. Confidence Tiers

| Tier | When |
|------|------|
| High | Direct evidence (file:line citations, multiple consistent examples) OR explicit derivation OR explicit user decision |
| Medium | Single example, inferred from related code, partially supported by upstream |
| Low | No evidence, contradictory examples, speculative — MUST appear in Open Questions |

## 9. Numbering on Re-Use

When updating a spec via `/refine update`:

- Add new requirements with the next sequential ID (FR-008 if FR-007 exists, regardless of any deleted IDs)
- Mark deleted requirements `[DELETED — <reason>]` and never reassign their IDs
- For ID reservations across contexts, check the artifact's most recent ID before assigning

## 10. Changelog Discipline

Every modification gets a Changelog entry:

| Date | Section | Change | Reason | Operation |

Multi-part changes (e.g., one update that touches three sections) get one Changelog row per discrete change.
